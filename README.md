# Service Observability SQL Demo

A compact analyst portfolio project showing how to monitor service quality, detect regressions, investigate root causes, and communicate operational recommendations from SQL-backed data.

This demo is intentionally aligned with Service Analyst / Operations Analyst roles where the job is to own service visibility across clients, SLAs, system behavior, and cross-functional follow-up.

## What this demonstrates

| Capability | Evidence in this repo |
|---|---|
| SQL analysis | `sql/*.sql` queries against a realistic SQLite service dataset |
| SLA / SLO monitoring | Client and fleet-level accuracy, timeliness, and completion metrics |
| Regression detection | Rolling-window comparisons and severity flags |
| Root-cause analysis | Issue-category breakdowns by client, robot, store, and workflow stage |
| Capacity planning | Labeling workload forecast based on contracted accuracy and volume |
| Auditability | Generated issue follow-up log and service observability handbook |
| Communication | Markdown and HTML reports with concise analyst recommendations |

## Scenario

Simulated retail robotics service operations:

- autonomous robots scan retail stores,
- service commitments require accurate and timely deliverables,
- labeling capacity affects SLA performance,
- subtle regressions can appear by client, robot firmware, store, or workflow stage,
- analysts need to detect gaps, document findings, and hold issues through resolution.

The data is synthetic, but the workflow mirrors real service operations analysis.

## Repository map

```text
data/
  generate_sample_data.py      # creates synthetic observations, deliverables, issues, and labeling demand
sql/
  01_service_health.sql        # fleet/client SLA scorecard
  02_regression_detection.sql  # rolling regression checks
  03_root_cause_patterns.sql   # systemic issue patterns
  04_capacity_forecast.sql     # labeling capacity forecast
scripts/
  run_analysis.py              # builds SQLite db, runs SQL, writes reports
outputs/
  generated_report.md          # generated by scripts/run_analysis.py
  service_health_dashboard.html
  *.csv
docs/
  service_observability_handbook.md
  issue_audit_log_template.md
.github/workflows/ci.yml       # runs the demo and verifies outputs
```

## Quick start

```bash
python3 scripts/run_analysis.py
```

Open the generated dashboard:

```bash
python3 -m webbrowser outputs/service_health_dashboard.html
```

Or inspect the generated analyst report:

```bash
sed -n '1,220p' outputs/generated_report.md
```

## Example analyst questions answered

1. Which clients are below accuracy or timeliness SLA this week?
2. Did any client, firmware version, or robot group regress versus the prior period?
3. Which issue categories explain most missed commitments?
4. What labeling capacity is required next week if volume grows?
5. Which findings require engineering, operations, or client-success follow-up?

## Why this matters

A good service analyst does more than build a chart. They maintain a trusted operating system for service quality:

- define the source of truth,
- measure commitments consistently,
- detect subtle regressions,
- document root cause and follow-up,
- communicate clearly to technical and non-technical partners,
- verify that quality improved after changes ship.

This repo shows that workflow end-to-end in a small, reviewable project.
